# 无穷云 IDC 系统开发与注释规则

## 目标

这份规则不是为了“写得好看”，而是为了保证：

- 中断后能继续开发
- 换编辑器后能继续开发
- 换协作者后能继续开发
- 换 AI 对话后能继续开发

## 注释规则

### 什么时候必须写注释

下面这些情况，必须写清楚注释：

- 外部接口字段语义不直观
- 业务状态流转有约束
- 有兼容逻辑、降级逻辑或兜底逻辑
- 当前实现是为了绕开某个已知限制
- 某段逻辑和旧系统行为对齐
- 某个字段、方法、分支如果不解释，接手的人看不出意图

### 注释应该写什么

优先写“为什么”和“约束”，不要只写“代码正在做什么”。

好注释应该能回答：

- 这段逻辑解决什么业务问题
- 为什么不能用更简单的实现
- 它依赖哪个外部系统或历史约定
- 未来改动这里时要注意什么

### 注释不该写什么

- 不要写纯废话注释
- 不要把代码逐行翻译成中文
- 不要让注释和实际逻辑脱节

## 文档同步规则

以下情况必须同步更新仓库文档：

- 新增模块边界
- 改变目录职责
- 接入新的第三方或上游接口
- 改变当前开发主线
- 废弃旧实现
- 改变下一阶段优先级

最少更新这些文件之一：

- [`docs/current-state.md`](/Users/a1/Documents/codex/bysat-idc/docs/current-state.md)
- [`docs/project-map.md`](/Users/a1/Documents/codex/bysat-idc/docs/project-map.md)
- [`README.md`](/Users/a1/Documents/codex/bysat-idc/README.md)

## 续开发规则

每次准备结束一个阶段时，要确保仓库里能回答下面这些问题：

- 当前唯一 active workspace 是什么
- 现在系统跑在哪些端口
- 当前真实对接到了哪里
- 哪些旧目录已经废弃
- 下一步应该先做什么
- 哪些敏感信息不能提交

如果这些问题无法只通过仓库内容回答，说明 continuity 还不够完整。

## Git 规则

- 持续把有效上下文同步到 git 跟踪文件，不要依赖聊天记录。
- 不要使用 git submodule 来保存这种上下文，普通文档文件就够了。
- 敏感配置放本地 `.env.local`，不要进仓库。
- 重大整理、架构切换、接口对接完成后，要有清晰提交信息。
